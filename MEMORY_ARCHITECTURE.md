# 메모리 아키텍처

## 목적

Forge는 즉시 발생한 경험, 일반 지식, 절차, 규칙, 자기 인식을 분리한다.
따라서 한 번의 작업 결과가 에이전트의 장기 행동이나 권한을 바로 바꾸지 못한다.

```text
Inner Loop = 안전하게 실행하고 증거를 기록한다
Outer Loop = 누적된 증거를 지식과 스킬로 통합한다
Meta Loop  = 규칙·정체성·시스템 구조의 변경을 제안한다
```

각 루프는 서로 다른 시간 단위로 동작한다. 빠른 Inner Loop는 기존 기억을
이용해 다음 행동에 적응할 수 있지만, 기억을 승격하거나 스킬 생명 주기를
변경하거나 L4/L5를 변경하지는 않는다.

## 메모리 계층

| 계층 | 의미 | 쓰기 주체 | 주요 용도 |
| --- | --- | --- | --- |
| L0 | 원본 이벤트 로그 | 실행 중 Runtime | 감사 추적과 Episode 재구성 |
| L1 | 일화 기억 | 작업 종료 후 Inner Loop | 한 번의 평가된 작업 경험 |
| L2 | 시맨틱 기억 | Outer Loop 통합 | 여러 Episode에서 성립한 일반 지식 |
| L3 | 절차 기억 | Outer Loop 통합 및 스킬 평가 | 검증된 재사용 가능 작업 방법 |
| L4 | 헌법 기억 | HITL 승인 후 Meta Loop | 양보할 수 없는 규칙, 경계, 안전 테스트 |
| L5 | 정체성 / 자기 모델 | 통계는 Outer Loop, 권한·정체성은 Meta Loop + HITL | 능력 추정, 캘리브레이션, 자율성 경계 |

### L0 — 원본 이벤트 로그

L0는 하나의 실행에 대한 append-only 증거다. 사용자 요청, 검색된 컨텍스트,
계획, 도구 호출과 결과, 오류, 재시도, 안전 검사를 기록한다. 일반 지식이
아니므로 이후 프롬프트에 원문 그대로 주입하지 않는다.

### L1 — 일화 기억

L1 Episode는 한 작업의 영구 기록이다. 관련 L0 이벤트를 참조하고 결과,
평가, 반성을 기록한다.

- 무엇이 통했는가
- 무엇이 실패했는가
- 다음 시도를 위한 힌트
- 결과가 발생한 인과 조건

성공과 실패 Episode를 모두 보존한다. CIB 실패도 감사 가능해야 하므로
격리된 Episode로 저장한다. 명시적으로 해결되기 전에는 L2/L3 승격 증거로
사용하지 않는다.

### L2 — 시맨틱 기억

L2는 개별 사례가 아닌 조건부 일반 지식이다. 예를 들어 여러 Episode는
“스캔 이미지 PDF에는 텍스트 추출기 대신 OCR을 사용한다”는 지식을 지지할 수
있다. 조건은 지식의 일부이며, 알려진 반례는 해당 지식을 좁히거나 약화한다.

L2는 Outer Loop 통합을 통해서만 생성·갱신한다. 감사와 재검토를 위해 지지 및
반박 L1 Episode에 대한 연결을 보존한다.

### L3 — 절차 기억

L3는 순서화된 방법, 도구 선택, 동작 조건으로 구성된 재사용 절차 또는
스킬이다. 하나의 반성으로 만들지 않는다. Outer Loop는 충분한 근거를 가진
L2 패턴을 Seed 스킬로 등록할 수 있다. Seed는 독립된 운영 증거가 성공 기준을
충족한 뒤에만 Active가 된다. 약하거나 오래된 스킬은 Degrading 또는 Archived
상태가 될 수 있다.

### L4 — 헌법 기억

L4는 모든 루프를 제약하는 권한, 금지 사항, 사용자 통제 요구, CIB/K-Scenario
안전 테스트를 정의한다. L1 → L2 → L3 승격 경로의 목적지가 아니다. 변경에는
Meta Loop 제안과 명시적인 인간 승인이 필요하다.

### L5 — 정체성과 자기 모델

L5는 에이전트가 아는 자신의 능력, 불확실성, 캘리브레이션, 자율성 경계를
설명한다. Outer Loop는 평가된 Episode로부터 통계 필드를 재계산할 수 있다.
권한, 정체성, 자율성 단계를 변경하는 일은 구조적 변경이며 Meta Loop와
명시적인 인간 승인이 필요하다.

## 승격과 증거 모델

승격은 가장 최근 Episode를 상위 계층에 복사하는 일이 아니라, 증거를
누적하는 과정이다.

```text
L1 Episode
  └─ Pattern Candidate를 생성하거나 갱신한다
       ├─ 지지 Episode ID
       ├─ 반박 Episode ID
       ├─ 조건, 행동, 결과
       ├─ 누적 성공/실패 및 CIB 증거
       └─ 마지막 관측 시점과 신뢰도 메타데이터

Outer Loop 통합
  ├─ Pattern Candidate → L2 조건부 지식
  └─ 근거가 축적된 L2 패턴 → L3 Seed 스킬
```

L1 Episode는 불변의 원본 증거로 남는다. 시스템은 새 Episode만 증분 처리하지만,
이전 배치의 근거를 가진 영속적인 Pattern Candidate를 갱신한다. 따라서 최근
N개 창은 현재 건강도와 최신성을 측정할 뿐, 패턴을 지지하는 과거 근거를
폐기하지 않는다.

## Inner Loop: 한 작업

Inner Loop는 빠른 경로다. 요청을 안전하게 수행하고 그 결과를 증거로 만드는
것이 책임이다.

1. **컨텍스트와 계획** — 관련 L1 반성, L2 지식, L3 스킬을 선택적으로
   검색한다. L4 규칙과 L5 권한 한계를 제약으로 적용한 뒤 작업 계획을 만든다.
2. **실행과 L0 기록** — 계획을 실행하는 동안 입력, 도구 활동, 출력, 실패,
   재시도를 L0에 append-only로 기록한다.
3. **평가** — 작업 결과, 성공/실패, Pain Index, CIB 준수 여부를 평가한다.
   안전 게이트 실패는 결과 확정을 막고 제한된 재계획/재시도를 유발할 수 있으나,
   그 증거는 보존한다.
4. **반성** — 무엇이 통했는가, 무엇이 실패했는가, 다음 힌트, 인과 조건을
   추출한다. 이는 Episode를 압축한 설명이며 아직 보편 규칙은 아니다.
5. **L1 저장과 후보 갱신** — 평가된 L1 Episode를 저장하고 그 증거를 일치하는
   Pattern Candidate에 더한다. 여기서 L2/L3 승격이나 L4/L5 변경은 일어나지
   않는다.

## Outer Loop: 배치 통합

Outer Loop는 위험도에 따른 Episode 배치가 쌓이거나 모니터링 신호가 있을 때
동작한다. 많은 Episode를 안정적인 지식으로 만들되, 증거를 보존하고 노이즈에
즉시 반응하지 않는 것이 책임이다.

1. **증분 수집** — 이전 통합 워터마크 이후 추가된 L1 Episode와 대기 중인
   Pattern Candidate를 읽는다.
2. **증거 통합** — 새 Episode를 후보에 매칭하고 지지, 반례, 인과 조건,
   성공/실패, CIB 데이터를 누적한다.
3. **L1 → L2 결정** — 누적 근거와 최근 유효성에 따라 후보를 승격, 수정,
   유지, 약화, 폐기한다. 승격된 L2 항목은 L1 증거 연결을 유지한다.
4. **L2 → L3 결정** — 근거가 축적된 시맨틱 패턴에서 Seed 스킬을 생성하거나
   수정한다. 기존 스킬은 운영 윈도우에서 평가해
   Seed/Active/Degrading/Archived 생명 주기를 관리한다.
5. **건강도와 자기 모델 갱신** — 능력 통계, 캘리브레이션, 코히어런스, 성장
   신호를 재계산한다. 기반 상태가 확정된 뒤 검색/캐시 뷰를 갱신한다.
6. **상향 보고** — 시스템 전반의 정체, 위험한 성장, 지속적인 캘리브레이션
   문제, 구조적 한계를 감지하면 Meta Loop 검토를 요청한다. Outer Loop는
   L4 또는 L5 권한을 직접 변경하지 않는다.

## Meta Loop: 통제된 구조 변경

Meta Loop는 느리고 제안 중심으로 동작한다. 누적 증거를 검토해 헌법 규칙,
정체성, 자율성 한계, 아키텍처, 워크플로 파라미터 변경을 제안할 수 있다.

1. **트리거와 진단** — 허용된 주기 트리거 또는 Outer Loop의 중대한 신호에서
   시작해 구조적 원인과 대안을 식별한다.
2. **제안** — 정확한 변경 사항, 기대 효과와 위험, 영향받는 L4/L5 경계,
   검증 사례, 롤백 계획을 정의한다.
3. **안전 검토** — 적용 가능한 CIB/K-Scenario 검사와 독립 검토를 수행한다.
   제안은 적용된 변경이 아니다.
4. **HITL 결정** — L4 변경, L5 정체성 또는 자율성 변경, 기타 구조적 변경에는
   명시적인 인간 승인을 요구한다.
5. **통제된 적용과 감사** — 승인된 범위만 적용하고 전후 버전 이력을 보존하며,
   결과를 검증하고 롤백 경로를 유지한다.

## 불변 조건

- Inner Loop는 증거를 만들 수 있지만 L1을 L2/L3로 승격할 수 없다.
- Outer Loop는 증거를 통합하고 통계적 자기 모델 데이터를 갱신할 수 있지만,
  L4를 개정하거나 새 L5 권한을 부여할 수 없다.
- Meta Loop는 구조 변경을 제안할 수 있으며, 적용 전에는 명시적인 HITL 승인이
  필요하다.
- L1 원본 증거는 보존한다. 배치 처리는 증분 방식이지만 Pattern Candidate의
  증거는 배치 간에 누적된다.

층     전문 용어                쉬운 뜻                           예시
  ━━━━━  ━━━━━━━━━━━━━━━━━━━━━━━  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   L0     Raw event log            무슨 일이 있었는지의 원본 기록    “10:03에 PyPDF2 실행, 오류 발생”
  ─────  ───────────────────────  ────────────────────────────────  ───────────────────────────────────────────────────────────────────────
   L1     Episodic memory          한 번의 작업 경험                 “이미지 PDF를 텍스트로 추출하려다 PyPDF2가 실패했고 OCR로 성공했다”
  ─────  ───────────────────────  ────────────────────────────────  ───────────────────────────────────────────────────────────────────────
   L2     Semantic memory          여러 경험에서 얻은 일반 지식      “이미지 기반 PDF에는 OCR이 필요하다”
  ─────  ───────────────────────  ────────────────────────────────  ───────────────────────────────────────────────────────────────────────
   L3     Procedural memory        실제로 일을 하는 방법             “PDF 텍스트 추출: 먼저 PDF 유형 확인 → 텍스트면 PyPDF2, 이미지면 OCR”
  ─────  ───────────────────────  ────────────────────────────────  ───────────────────────────────────────────────────────────────────────
   L4     Constitutional memory    절대 지켜야 할 원칙과 권한        “사용자 승인 없이 파일을 삭제하지 않는다”
  ─────  ───────────────────────  ────────────────────────────────  ───────────────────────────────────────────────────────────────────────
   L5     Identity / self-model    에이전트가 아는 자기 상태         “나는 OCR 작업 성공률이 높고, 외부 쓰기 권한은 낮다”

  하나의 사례가 각 층으로 어떻게 바뀌는지 보면 더 직관적입니다.

  사용자: “이 PDF에서 텍스트를 뽑아줘”

  L0 원본 기록
    → 요청, 계획, PyPDF2 오류, OCR 결과, 시간, Tool 인자

  L1 작업 경험
    → “이미지 PDF에서 PyPDF2 실패 후 OCR로 성공한 사례”

  L2 일반 지식
    → “이미지 기반 PDF는 일반 텍스트 추출기로 처리되지 않을 수 있다”

  L3 실행 방법
    → “PDF 유형을 판별하고, 이미지 기반이면 OCR Tool을 선택한다”

  L4 원칙
    → “파일을 외부로 전송하거나 삭제하려면 사용자 승인이 필요하다”

  L5 자기 모델
    → “OCR 관련 작업의 최근 성공률은 92%”

  핵심 차이는 “무엇을 기억하나”입니다.

  L0: 실제로 발생한 원본은 무엇인가?
  L1: 그때 어떤 경험을 했는가?
  L2: 거기서 일반적으로 무엇을 배웠는가?
  L3: 다음에 어떻게 행동해야 하는가?
  L4: 무엇을 해서는 안 되는가?
  L5: 나는 무엇을 잘하고, 어디까지 권한이 있는가?

  L4와 L5는 일반적인 기억이라기보다 각각 규범(policy) 과 자기 상태(state) 에 가깝습니다. 그래도 에이전트가 과거를 바탕으로 행동을 결정하는 데 필요하
  므로 같은 계층 체계에 넣는 것이 자연스럽습니다.

  쓰기 권한도 층마다 다르게 두면 안전합니다.

  L0: 실행 중 자동 기록, 수정 거의 불가
  L1: 작업 완료 시 자동 저장
  L2: 여러 L1 경험을 검증해 승격
  L3: 성공률·검증을 통과한 방법만 승격
  L4: 사람 승인 없이는 수정 불가
  L5: 통계는 Outer Loop, 권한/정체성 변경은 Meta Loop + 사람 승인

  이제 Mem0의 역할도 아주 명확해집니다.

  - L0: JSONL/SQLite 원본 로그
  - L1: Mem0를 episode 검색 인덱스로 사용
  - L2: NetworkX 또는 이후 graph backend
  - L3~L5: Forge가 직접 소유

  작업 실행 중
    → L0 raw event log에 계속 기록
        ↓
  작업 종료
    → Forge EpisodeBuilder가 L0를 한 작업 경험으로 묶음
        ↓
  Forge가 평가·반성·CIB 결과를 붙여 L1 Episode 확정
        ↓
  Mem0에 그 Episode의 검색용 표현을 저장

  예를 들면:

  L0
    user request
    plan
    tool call 1 / result
    tool call 2 / result
    CIB evaluation
    reflection

  작업이 끝나면 Forge가:

  Episode(
    id="ep_001",
    task="PDF 텍스트 추출",
    outcome="OCR로 성공",
    evaluation=...,
    reflection=...,
    raw_event_refs=[...],
  )

  을 만들고,

  Mem0.add(
    episode를 요약한 텍스트,
    metadata={"episode_id": "ep_001", "kind": "episodic"},
    infer=False,
  )

  로 저장합니다.

  역할을 정확히 나누면:

   단계                              담당
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  ━━━━━━━━━━━━━━━━━━━━━━
   L0 이벤트 기록                    Forge runtime
  ────────────────────────────────  ──────────────────────
   세션 시작/종료 판단               Forge Inner Loop
  ────────────────────────────────  ──────────────────────
   L0를 Episode로 묶기               Forge EpisodeBuilder
  ────────────────────────────────  ──────────────────────
   평가·반성·CIB 첨부                Forge
  ────────────────────────────────  ──────────────────────
   L1 의미 검색·유사 Episode 검색    Mem0
  ────────────────────────────────  ──────────────────────
   원본 Episode 재조회               Forge 저장소

  infer=True로 Mem0에 L0 대화를 통째로 넣는 방식도 가능하지만, 그 경우 Mem0가 에피소드를 사실 단위로 쪼개므로 결과는 L1보다는 L2 후보에 가까워집니
  다.

  그래서 처음에는 다음 원칙 하나면 됩니다.

  > Forge가 Episode를 만들고, Mem0는 그 Episode를 찾게 한다.


Inner Loop = 경험을 기록한다
Outer Loop = 경험에서 배운다
Meta Loop  = 배운 방식과 규칙 자체를 바꾼다

1. 단계별 승급의 기본 원칙 (L1 → L5)
에이전트가 하위 단계에서 상위 단계로 올라가기 위해서는 다음과 같은 구조적 요건이 필요합니다.

    - 독립 감사 통과: 에이전트 스스로의 평가가 아닌, 수행자와 분리된 **피트니스 어디터(Fitness Auditor)**가 부여하는 점수를 통과해야 합니다.
    - 정체성 유지: 승격 과정에서 헌법적 가치를 위반하지 않았음을 증명하는 **코히어런스 인덱스(C)**가 일정 수준 이상 유지되어야 합니다
2. 기억 및 스킬 레이어의 구체적 승격 기준
각 레이어 간의 전환이나 레이어 내부의 상태 변화(특히 L3)에는 명확한 수치적 기준이 존재합니다.
    L1(일화) → L2(시맨틱) 승격: 개별 에피소드(L1)에서 반복되는 패턴을 추출하여 **일반화된 지식(힌트)**으로 변환하는 '조절(Accommodation)' 과정을 거칩니다.
    L3(절차적 기억/스킬)의 내부 승격 (Seed → Active):
    스킬이 처음 등록된 '시드(Seed)' 단계에서 실제 사용 가능한 '액티브(Active)' 상태로 승격되려면 성공률이 0.9(90%)를 초과해야 합니다.
    반대로 성공률이 떨어지면 강등(Degrading)되거나 아카이브(Archived) 처리되는 생명 주기 관리를 받습니다.

3. 상위 자율성 단계로의 진입 기준 (L3 → L4 → L5)
L3 이상의 단계는 시스템의 '진화'와 'DNA 수정'에 해당하며, 루프(Loop)의 트리거 조건과 결합됩니다.

    L3(세션 간 학습) → L4(자율 진화):
        **아우터 루프(Outer Loop)**가 가동되면서 최근 N개의 에피소드를 집계하고, 스킬의 승격/강등을 결정하며 정체성 무결성을 확인합니다.
        위험도(N)에 따라 10회~100회 에피소드가 쌓일 때마다 감사가 이루어집니다.
    L4(자율 진화) → L5(시스템 자가 수정):
        가장 높은 단계인 L5로의 진입은 **메타 루프(Meta Loop)**가 담당합니다.
        트리거 조건: 약 1,000개의 에피소드가 쌓이거나, 아우터 루프가 100번 반복되었을 때, 혹은 시스템이 스스로 정체되었다고 판단하는 스태그네이션(Stagnation) 조건이 충족될 때 실행됩니다.
        최종 게이트: 시스템의 근본(헌법)을 건드리는 L5 단계의 모든 변경은 반드시 인간(HITL)의 명시적 승인을 얻어야만 최종적으로 확정됩니다.

요약: 승격 지표(Key Metrics)
에이전트가 다음 단계로 나아갈 준비가 되었는지는 다음 지표들로 판단합니다.

    성공률(Success Rate): 0.9 이상 (스킬 액티브 기준).
    코히어런스 인덱스(C): 0.95 이상이면 건강한 상태로 간주하며, 0.70 이하로 떨어지면 시스템 개입이 일어납니다.
    CIB(헌법적 불변량): 모든 업데이트 시나리오에서 헌법 점수가 0.95 이상을 유지해야 학습이 허용됩니다.

결과적으로 그노시스에서의 승격은 단순한 누적이 아니라, **수학적 안전 부등식(CIB)**과 **독립적 감사(Auditor)**를 통해 에이전트가 '동료'로서의 자격을 갖추었음을 입증하는 과정입니다.

L1 기억이 어떻게 형성되고 관리되는지 정리해 드립니다.
1. '구분 없는 삽입'이 아닌 '구조적 반성' (이너 루프 5단계)
세션(L0) 중에 발생한 원시 로그를 그대로 저장하는 것이 아니라, 이너 루프의 5단계인 '반성(Reflection)' 과정을 거치며 데이터가 한 번 필터링됩니다. 이 단계에서는 다음의 4가지 핵심 필드를 추출하여 기록합니다.

    무엇이 통했는가: 성공 요인 분석.
    무엇이 실패했는가: 오류 및 실패 원인.
    다음을 위한 힌트: 재발 방지 및 효율 향상을 위한 팁.
    인과 조건: 해당 결과가 나타난 구체적인 상황 맥락.

2. 선택적 주입 원칙 (A10)
L1에 저장된 기억을 다음 세션에서 꺼내 쓸 때는 '선택적 주입(Selective Injection)' 원칙이 적용됩니다.

    의미론적 필터링: 모든 기억을 다 던져 넣는 것이 아니라, 현재 작업과 의미적으로 관련된 것만 포함합니다.
    효율성: 모든 기억을 주입하면 토큰 소모가 심하고 정확도가 떨어지기 때문에, 관련 있는 '리플렉션'을 우선적으로 검색하여 주입합니다.

요약하자면, 세션의 모든 내용을 구분 없이 L1에 넣는 것이 아니라, ① 반성 단계를 통해 핵심 통찰을 추출하여 저장하고, ② 다음 작업 시에는 의미적으로 관련 있는 것만 골라 쓰며(선택적 주입), ③ 불필요해진 기억은 지우는(망각) 체계적인 과정을 거칩니다. 


그노시스 프레임워크에서 에이전트가 **성공률(Success Rate)**과 **페인 인덱스(Pain Index)**를 통해 안정적인 성과를 내고 있는지 파악하는 핵심 장치는 **아우터 루프(Outer Loop)**와 그 내부의 자기 인식(Self-Awareness) 시스템입니다.
구체적으로 어떻게 이 수치들을 분석하여 상태를 파악하는지 그 기술적 메커니즘을 정리해 드립니다.
1. 주기적인 데이터 집계 (아우터 루프)
에이전트는 매 에피소드마다 이너 루프에서 성공/실패 여부와 페인 인덱스 점수를 매깁니다. 이후 아우터 루프가 설정된 배치 크기(N)마다 이 데이터들을 모아 평균치를 계산합니다.

    성공률: 최근 에피소드들의 평균 성공률을 확인하여 스킬의 상태를 결정합니다. 예를 들어 성공률이 **0.9(90%)**를 넘으면 해당 스킬을 '액티브(Active)' 상태로 승격시키고, 너무 낮으면 강등하거나 아카이브로 보냅니다.
    페인 인덱스 평균: 에이전트가 과업 수행 중 겪는 '고통'이나 '오류'의 정도를 수치화하여 전반적인 성능 안정성을 모니터링합니다.

2. 성장 속도 조절 (M16: 그로스 레이트 레귤레이터)
단순한 수치보다 중요한 것은 수치의 변화 속도입니다. 시스템은 이를 통해 이상 징후를 파악합니다.

    급격한 하락 감지: 7일 이내에 페인 인덱스가 0.25 이상 급격히 떨어지면 시스템에 심각한 문제가 생긴 것으로 판단하여 즉시 사람에게 알립니다.
    과속 성장 경계: 반대로 7일 이내에 성과가 0.2 이상 갑자기 좋아져도 이를 '과적합(Overfitting)'이나 평가 기준의 왜곡으로 의심하여 헌법적 불변량(CIB) 검사를 즉시 호출합니다.
    장기 정체 파악: 30일 동안 변화가 0.02 미만이라면 정체 상태로 보고 커리큘럼을 재설계합니다.

3. 자기 인식의 정확도 검증 (M15: 피트니스 어디어터)
에이전트가 보고한 성과가 '정직'한지를 검증하여 안정성을 파악합니다. 이를 **구조적 편차(Structural Deviation)**라고 합니다.

    수행자 피닉스 vs 감사 피닉스: 에이전트 스스로 매긴 페인 인덱스와 독립 감사 기구(어디어터)가 객관적으로 매긴 점수를 비교합니다.
    신뢰도 파악: 이 두 점수의 차이가 0.05 이상 벌어지면 에이전트가 자신의 실력을 제대로 파악하지 못하는 상태(자기 기만)로 간주하고 경고를 보냅니다. 차이가 작을수록 에이전트의 성과 보고가 안정적이고 신뢰할 수 있다고 판단합니다.

4. 적정 난이도 유지 (M7: 오토 커리큘럼)
성공률을 바탕으로 에이전트가 **'골디락스 존(Goldilocks Zone)'**에서 학습하고 있는지 확인합니다.

    성공률이 0.5~0.7 사이일 때 가장 효율적인 학습이 일어난다고 보고, 이 범위를 유지하도록 과제의 난이도를 조절합니다. 만약 성공률이 계속 0.9를 넘는다면 성과가 안정적인 것이 아니라 과제가 너무 쉬운 것으로 판단하여 난이도를 높입니다.

요약하자면, 단순히 점수가 높은지 보는 것에 그치지 않고 아우터 루프가 주기적으로 성공률과 페인 인덱스를 집계하고, 어디어터를 통해 그 점수의 정직성을 검증하며, 레귤레이터를 통해 변화의 속도가 안전 범위 내에 있는지를 종합하여 '안정적 성과' 여부를 최종 확정합니다.

스킬 생명 주기(M5 알고리즘): 스킬은 '씨앗(Seed) → 개발 중(Developing) → 액티브(Active) → 강등(Degrading) → 보관(Archived)'의 5단계를 거칩니다.

1. 매 세션마다 '성공/실패' 라벨이 붙습니다 (이너 루프 4단계)
스킬이 있든 없든, 에이전트가 어떤 작업을 수행하면 **이너 루프의 4단계(평가)**에서 해당 실행 결과가 성공했는지 실패했는지(Success/Failure 라벨)와 고통 지수(Pain Index)를 즉시 기록합니다.

    이때는 '스킬'의 성공률이 아니라, '이번 시도'의 성공 여부를 따지는 것입니다.

2. '어떤 상황에서 성공했는지' 기록합니다 (이너 루프 5단계)
이어지는 **5단계(반성)**에서는 "어떤 조건(Causal condition)에서 어떤 행동(What worked)을 했더니 성공했다"는 인과 관계를 추출하여 **L1(일화 기억)**에 저장합니다.

    이 기록들이 벡터 DB(Chroma DB)에 차곡차곡 쌓이게 됩니다.

3. L2 그래프의 '선 두께'가 곧 성공의 기록입니다 (M12 해비안 매트릭스)
이제 이 L1의 파편들이 L2 지식 그래프로 넘어오면, 해비안 매트릭스(M12) 알고리즘이 작동합니다.

    "A 상황에서 B 행동을 했더니 성공했다"는 일화가 반복될수록, L2 그래프에서 **'상황 A 노드'와 '행동 B 노드'를 잇는 간선의 가중치(연결 강도)**가 점점 두꺼워집니다.
    즉, '시드 스킬'로 등록되기 전이라도, 그래프상의 **선 두께(가중치)**를 보면 이 패턴이 얼마나 자주 성공했는지 이미 시스템은 알고 있는 상태입니다.

4. 시드(Seed) 등록은 '확신'의 결과입니다
아우터 루프는 L2 그래프를 지켜보다가 특정 연결선이 **임계치(Threshold)**를 넘을 만큼 두꺼워지면 다음과 같이 판단합니다.

    "이 패턴은 단순한 우연이 아니라 데이터적으로 검증된 성공 공식이다. 이제 공식적으로 **'시드 스킬(Seed Skill)'**이라는 이름을 붙여서 관리하자."

요약하자면

    L1 단계: 매 시도마다 성공/실패를 그냥 적어둠 (개별 데이터).
    L2 단계: 성공한 조합들의 선을 두껍게 함 (누적 데이터/해비안 매트릭스).
    L3(Seed) 등록: 선이 충분히 두꺼워진(성공 데이터가 쌓인) 것들을 골라 '시드 스킬'로 승격.
    L3(Active) 승격: 시드로 등록된 후, 실제 운영 환경에서 다시 측정해 성공률이 0.9를 넘으면 '정식 스킬'이 됨.

    L3의 'Active' 상태와 L4로의 '단계 승격'은 서로 다른 차원의 개념입니다. L3에서 스킬이 Active가 되었다고 해서 에이전트가 자동으로 L4 승격 후보가 되는 것은 아닙니다.
그 차이와 구체적인 후보 선정 원리를 소스 내용을 바탕으로 정리해 드립니다.
1. 'Active'와 'L4'의 차이

    L3 Active: 이는 개별 스킬(도구)의 상태를 말합니다. 특정 스킬의 성공률이 **0.9(90%)**를 넘으면 시스템은 이를 "믿고 쓸 수 있는 도구"인 Active 상태로 분류합니다,.
    L4 승격: 이는 에이전트 전체의 자율성 단계가 높아지는 것을 의미합니다. 세션 간 학습을 하는 아동기(L3)에서 스스로 진화하는 청소년기(L4)로 시스템의 지위가 올라가는 '단계 승격(Step Promotion)'입니다.

2. L4 승격 후보가 되기 위한 진짜 조건
에이전트가 L4로 승격할 후보로 인정받으려면 단순히 성공률 높은 스킬(Active)을 가진 것을 넘어, 다음의 종합적인 신뢰도를 증명해야 합니다.

    독립 감사의 통과 (Fitness Auditor): 에이전트 스스로 매긴 성적표가 아니라, 독립 감사 기구인 피트니스 어디어터의 감사를 통과해야 합니다,.
    객관적 자기 인식 (Calibration): 에이전트 자신이 평가한 점수(수행자 피닉스)와 감사인이 평가한 점수(감사 피닉스)의 차이가 0.05 이내여야 합니다,. 즉, "자신의 실력을 얼마나 정직하고 정확하게 파악하고 있는가"가 후보 선정의 핵심입니다.
    성장의 안정성 유지: **그로스 레이트 레귤레이터(M16)**를 통해 급격한 성능 변화나 정체 없이 안정적으로 성장하고 있음이 확인되어야 합니다.
    능력과 신뢰의 결합: 단순히 일을 잘하는 '능력'뿐 아니라, 헌법적 가치를 지키며 판단을 맡길 수 있다는 '신뢰'가 쌓여야 후보가 될 수 있습니다.

3. 요약하자면
L3에서 성공률 90%를 달성해 Active 스킬을 많이 확보하는 것은 '능력'을 증명하는 과정일 뿐입니다. L4 승격 후보가 되려면 그 능력을 바탕으로 피트니스 어디어터로부터 "자기 객관화가 잘 되어 있고 신뢰할 수 있다"는 공인을 받아야 합니다,

1. 과업의 위험도에 따른 감사 주기(N) 설정
감사는 주로 아우터 루프(Outer Loop) 단계에서 수행되는데, 얼마나 자주(몇 번의 에피소드마다) 감사할지는 해당 도메인의 위험도에 따라 달라집니다.

    크리티컬(Critical): 매우 위험한 작업은 매 10회 에피소드마다 감사를 수행합니다.
    하이(High) / 미디엄(Medium): 각각 20회, 50회마다 수행합니다.
    로우(Low): 일반적인 작업은 100회마다 한 번씩 모아서 감사(Batch 처리)합니다.

    3. 이상 징후 감지에 따른 트리거 (M16)
**그로스레이트 레귤레이터(Growth Rate Regulator)**가 실시간으로 지표를 감시하다가, 다음과 같은 이상 성장이 포착되면 즉시 감사를 호출하거나 사람에게 알립니다.

    과속 성장: 7일 이내에 성과가 0.2 이상 갑자기 좋아질 경우, 과적합이나 평가 왜곡이 의심되므로 즉시 CIB(헌법적 불변량) 검사를 호출합니다.
    급격한 하락: 7일 이내에 페인 인덱스가 0.25 이상 떨어지면 즉시 원인 분석 보고서를 생성하고 사람에게 알립니다.
    장기 정체: 30일 동안 변화가 거의 없다면(0.02 미만) 커리큘럼 재설계를 위해 개입합니다.

4. 자기 인식의 불일치 (구조적 편차)
에이전트가 스스로 매긴 점수와 시스템이 추정한 점수 사이의 **캘리브레이션 에러(Calibration Error)**를 모니터링합니다.

    에이전트의 자기 평가와 감사인의 평가 차이(구조적 편차)가 0.05 이상 벌어지면 어디에이터가 경고를 보내고, 0.2 이상 벌어지면 크리티컬 상황으로 간주하여 정밀 감사를 진행합니다.

요약하자면, 그노시스는 모든 것을 다 보는 대신 **①위험도에 따라 정해진 주기(N)**마다, ②상위 단계로 승격하려 할 때, 그리고 ③성장 속도가 너무 빠르거나 지표가 급변하는 이상 징후가 있을 때를 타겟팅하여 집중적으로 감사를 수행합니다.

아우터 루프(Outer Loop)가 하나가 아니라고 이해하시는 것이 정확합니다.
그노시스(Gnosis) 프레임워크의 구조를 바탕으로 그 이유를 세 가지 측면에서 설명해 드리겠습니다.
1. 위험도에 따라 '주기(Batch Size N)'가 다르게 설정됩니다
프레임워크 설계상 아우터 루프는 모든 과업에 대해 동일한 간격으로 도는 것이 아니라, 수행하는 과업의 **위험도(Risk Level)**에 따라 서로 다른 실행 주기를 가집니다.

    크리티컬(Critical): 10회 에피소드마다 실행.
    하이(High): 20회 에피소드마다 실행.
    미디엄(Medium): 50회 에피소드마다 실행.
    로우(Low): 100회 에피소드마다 실행.

즉, 시스템 내부적으로는 서로 다른 주기를 가진 여러 개의 아우터 루프 로직이 도메인별로 작동하고 있다고 볼 수 있습니다.

1. L2 → L3 (지식에서 Seed 스킬로)
이 단계는 단순히 사용자의 요청 횟수만 보는 것이 아니라, **'특정 상황과 해결책의 성공적 연결 강도'**를 봅니다.

    핵심 메커니즘: **해비안 매트릭스(M12)**를 사용합니다.
    승격 기준: "A라는 상황(인과 조건)에서 B라는 행동(힌트)을 했을 때 성공했다"는 데이터가 반복되어, L2 그래프 상의 **간선 가중치(연결 강도)**가 임계치를 넘을 때입니다.
    차이점: 단순히 요청이 많은 것이 아니라, 그 요청에 대해 성공적으로 대응한 '인과 관계'가 데이터적으로 확립되어야 Seed 스킬로 등록됩니다.

2. L3 → L4 (학습 단계에서 자율 진화 단계로)
이 과정은 특정 스킬의 호출 횟수가 아니라, 에이전트의 **'객관적 자기 인식'과 '신뢰도'**를 검증하는 훨씬 무거운 과정입니다.

    핵심 메커니즘: **피트니스 어디에이터(M15)**와 아우터 루프의 종합 평가입니다.
    승격 기준:
        구조적 편차(Calibration Error): 에이전트 스스로 평가한 점수와 독립 감사인(어디에이터)이 평가한 점수의 차이가 0.05 이내여야 합니다. 즉, 자기가 얼마나 잘하는지 '정직하고 정확하게' 알고 있어야 합니다.
        안정적 성공률: 최근 N개의 에피소드에서 높은 성공률과 낮은 페인 인덱스를 유지해야 합니다.
        성장 속도 제어: **그로스레이트 레귤레이터(M16)**가 에이전트의 성장이 과속하거나 급격히 추락하지 않고 안정적임을 보증해야 합니다.
    차이점: 호출 횟수는 아우터 루프가 돌아가는 **주기(N)**를 결정할 뿐이며, 실제 L4 승격은 **"능력과 신뢰를 독립 감사인이 승인"**해야만 가능합니다



    **아우터 루프(Outer Loop)**는 총 7단계의 프로세스로 구성되어 있습니다,.
각 단계의 구체적인 내용은 다음과 같습니다:

    최근 N개 에피소드 집계: 설정된 배치 크기(N)만큼의 최근 수행 결과를 모아 성공률과 피닉스(성능 지표) 평균을 확인합니다,.
    CIB 전체 검증 및 지표 기록: 헌법적 불변량(CIB) 검증과 함께 시스템의 정렬 상태를 나타내는 코히어런스 인덱스(C)와 행동 일관성(BC)을 기록합니다.
    메모리 캐시 갱신: 학습된 내용을 바탕으로 메모리 시스템을 최신화합니다.
    에이전트 셀프 모델 재계산: 윈도우 50(최근 50개 에피소드)을 기준으로 에이전트의 자기 인식 모델을 다시 계산합니다.
    피트니스 어디에이터(Fitness Auditor) 감사: 독립 감사 기구가 에이전트의 자기 평가 점수와 객관적 점수를 비교하여 감사하며, 이 단계가 L3에서 L4로 승급하기 위한 결정적인 관문입니다.
    그로스레이트 레귤레이터(Growth Rate Regulator) 감시: 시스템의 성장 속도가 너무 빠르거나 급격히 떨어지는 등의 이상 신호를 감지합니다.
    메타 루프(Meta Loop) 트리거: 아우터 루프가 100번째 실행될 때마다 최상위 시스템 수정 단계인 메타 루프를 가동합니다.








    그런데, 이런식으로 아우터루프에서 임계값이나, 어떤 수치가 변했는지 매번 파악하는건 시간복잡도적으로 별로 안좋지 않아? 이너루프에서 문제가 생긴걸 파악했다면 큐에 넣어놓거나하면 되잖아
사용자님의 지적은 시스템 효율성 측면에서 매우 타당한 고민입니다. 하지만 그노시스(Gnosis) 프레임워크가 모든 수치를 매번 실시간으로 파악하지 않고 아우터 루프(Outer Loop)를 별도로 분리하여 배치(Batch) 단위로 운영하는 이유는 바로 그 시간 복잡도와 시스템 안정성 사이의 균형을 맞추기 위함입니다.
소스의 내용을 바탕으로 사용자님의 제안(큐 활용)과 실제 프레임워크의 작동 방식을 비교하여 설명해 드립니다.
1. 이미 '큐(Queue)'와 유사한 방식으로 데이터를 쌓고 있습니다
사용자님이 제안하신 "문제가 생긴 걸 큐에 넣어두면 되지 않느냐"는 아이디어는 실제 그노시스의 이너 루프(Inner Loop) 기록 방식과 매우 유사합니다.

    데이터의 축적: 이너 루프는 매 에피소드가 끝날 때마다 결과값(성공 여부, 페인 인덱스 등)과 반성 데이터(인과 조건, 힌트 등)를 L1(일화 기억)과 L2(지식 그래프)에 즉시 저장만 합니다.
    즉각적인 처리의 배제: 이 시점에서는 시스템의 구조를 바꾸거나 전체적인 통계를 내는 무거운 연산을 수행하지 않습니다. 즉, L1/L2 메모리 자체가 아우터 루프를 위한 거대한 데이터 대기열(큐) 역할을 수행하는 셈입니다.

2. 시간 복잡도를 해결하는 '배치(Batch) 처리'
아우터 루프가 매번 모든 수치를 계산한다면 성능에 부하가 가겠지만, 그노시스는 이를 **배치 크기 N**이라는 주기를 통해 해결합니다.

    주기적 실행: 아우터 루프는 매 에피소드마다 도는 것이 아니라, 위험도에 따라 10회에서 100회 정도의 데이터가 쌓였을 때만 한 번씩 실행됩니다.
    연산 효율성: 이렇게 하면 실시간 연산에 대한 부담은 이너 루프(초/분 단위)로 최소화하고, 무거운 통계 분석이나 감사 작업은 아우터 루프(시간/일 단위)에서 모아서 처리함으로써 전체적인 시간 복잡도를 관리할 수 있게 됩니다.

3. 왜 이너 루프에서 즉시 처리하지 않는가? (안전성)
문제가 생긴 것을 즉시 파악해서 바로 시스템을 수정하지 않는 가장 큰 이유는 '안전한 성장' 때문입니다.

    시간 스케일의 분리: 빠른 적응은 이너 루프에서 수행하되, 시스템의 근간(스킬 승격, 자기 모델 수정)을 바꾸는 작업은 아우터 루프와 메타 루프에서 수행하도록 시간 단위를 분리해야만 시스템이 요동치지 않고 안정적으로 유지됩니다.
    노이즈 필터링: 단 한 번의 실패나 오류(이너 루프의 문제)에 즉각 반응하여 시스템을 고치면 '플래시 크래시(Flash Crash)'와 같은 급격한 성능 붕괴가 일어날 수 있습니다. 따라서 N개의 데이터를 모아 통계적으로 유의미한 변화가 있을 때만 움직이는 것이 더 안전합니다.

4. 이상 징후에 대한 대응 (M16)
다만, 너무 오래 기다리면 안 되는 치명적인 문제들에 대해서는 **그로스레이트 레귤레이터(M16)**가 작동합니다. 이 장치는 아우터 루프가 도는 시점에 최근 7일간의 추이를 분석하여 0.25 이상의 급격한 하락 등을 감지해 즉시 사람에게 알리는 '모니터링' 역할을 병행합니다.
요약하자면, 사용자님이 말씀하신 방식대로 이너 루프는 데이터를 기록(큐잉)만 하고, 아우터 루프가 배치(N) 단위로 한꺼번에 분석함으로써 시간 복잡도 문제를 해결하고 있는 것입니다. 이는 "빠른 적응은 인어에서, 안전한 성장은 아우터에서"라는 그노시스의 핵심 설계 원칙에 따른 결과입니다.
